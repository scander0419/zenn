## はじめに

最近の自分は、いわゆる **Vibe Coder** でした。生成AIに指示すれば、それっぽいツールは作れる。実際、フックや夜間バッチなど少し込み入った自動化が絡むツール（[codex-dreaming](https://github.com/scander0419/codex-dreaming)）も形にできました。

でも、ある日ふと気づきました。**自分が作ったコードを、自分で説明できない。**

`read_bytes()` と `read_text()` の違いは？ この `break` はどのループを抜ける？ と聞かれると固まる。特にAIが書いた部分はそうです。「動いている」けれど「分かっていない」。この記事は、その状態を立て直すために試している方法のメモです。

## この記事でわかること

- 「動くけど説明できない」Vibe Coder のつまずき方
- 自分の基礎の穴を炙り出す **6問の自己診断**（題材は自分のコード）
- 教科書を頭から読むのではなく、**自分のコードの「逆引き辞書」として使う**学び方

## Vibe Coderの壁: 動くけど説明できない

AIと一緒に開発していると、手は進むのにこういう状態になりがちです。

- 全体は動く。でも各行が「なぜそう書くか」を説明できない
- エラーが出たとき、AIに貼る以外の打ち手が無い
- 「ここをこう変えたらどうなる？」に自信を持って答えられない

要は **読めて・直せて・説明できる** が伴っていない。作れること自体は強みなので、足りない半分をどう埋めるか、という話です。

## 自分のコードに6問答えてみる（自己診断）

題材は、自分のリポジトリにある `fix_encoding.py`。PowerShell などが非UTF-8（cp932 / UTF-16）で保存したファイルを、UTF-8に直すだけの小さなツールです。1ファイルで完結していて、基礎の宝庫でした。

中身の抜粋（実物）:

```python
for dirpath, dirnames, filenames in os.walk(root):
    dirnames[:] = [d for d in dirnames if d not in SKIP_DIRS]
    for name in filenames:
        if seen >= MAX_FILES:
            break
        p = Path(dirpath) / name
        ...
        data = p.read_bytes()
        if not data or _looks_utf8(data):
            continue
```

自分に出した問い（一部）:

1. `read_bytes()` はなぜ `read_text()` ではないのか？ そもそも `str` と `bytes` の違いは？
2. `SKIP_DIRS`（`set`）はなぜ `list` ではなく `set` なのか？
3. この `break` はどのループを抜ける？

結果は……ほぼ全滅でした。「分からない」「なんとなく」しか出てこない。でもこれが良くて、**答えられなかった場所が、そのまま自分の穴の地図になりました。**

以下、特に効いた2つを書きます。

## 穴の例1: str と bytes（文字化けの正体）

そもそもコンピュータは**数字（バイト）しか保存できません**。人間が読みたいのは「日本語」という**文字**。そこで「この文字＝この数字」という変換ルールが要ります。これが**文字コード**（UTF-8 / cp932 / UTF-16 など）です。

- `str` = 人間が読む文字そのもの（`"日本語"`）
- `bytes` = ディスクに保存される生の数字の列
- `encode` = str → bytes、`decode` = bytes → str

ここが核心でした。

```python
text = "日本語"
print(text.encode("utf-8") == text.encode("cp932"))  # False
```

**同じ文字でも、ルールが違えば別の数字になる。** だから保存時と違うルールで読むと文字化けやエラーになる。これが自分のツールが直そうとしていた現象そのものでした。

そして `read_bytes()` を使う理由も腑に落ちました。`read_text()` は「UTF-8だろう」と**ルールを勝手に仮定**してしまう。だからまず生の `bytes` で読み、本当のルールを推定してから直す——という設計だったわけです。自分のコードなのに、ここで初めて理解しました。

## 穴の例2: break はどのループを抜けるか

これは恥ずかしい話です。自分は「**外側のループ**」と即答して、間違えました。

正解は「**一番内側のループだけ**」。`break` は最も内側のループしか抜けません。

```python
for dirpath, dirnames, filenames in os.walk(root):  # 外側（フォルダごと）
    for name in filenames:                          # 内側（ファイルごと）
        if seen >= MAX_FILES:
            break    # 内側だけ抜ける。os.walk は次のフォルダへ進んでしまう
```

つまり `MAX_FILES` は「全体での上限」として機能していない。致命的ではないけれど、**自分の作ったコードの微妙な挙動を、自分で指摘できた瞬間**でした。これが地味に嬉しかった。AI任せだったコードに、初めて自分の目が入った感覚です。

## 学び方: 教科書を「逆引き辞書」にする

基礎をやり直すと決めても、入門書を頭から解くのは（自分には）退屈で続きません。そこで、こういうループに変えました。

1. 教科書で概念を**1つだけ**学ぶ（例: str と bytes）
2. すぐ**自分のコード**でその概念が使われている所を探し、「これだ」と確認する
3. その部分を**AIなしで小さく再実装**してみる

この回し方が自分には合っていました。教材が「他人の練習問題」ではなく「自分の作ったもの」なので、やる意味が腹落ちするし、理解できると素直に嬉しい。`fix_encoding.py` の縮小版を、何も見ずに書き直せたときは、ちゃんと前進した実感がありました。

## 注意・ハマりどころ

- **一度に全部理解しようとしない。** 1日1概念で十分。欲張ると続かない。
- **AIに「答え」を書かせず「先生」として使う。** 説明させる・採点させるのはOK、でもコードは自分で書く。じゃないとまた同じ「説明できないコード」が増える。
- **必ずしも競技プログラミングに行く必要はない**、と個人的には思っています。自分が欲しかったのは「自分のコードを読んで直せる」基礎で、それには競プロより**自分の成果物が一番の教材**でした（目的次第なので、アルゴリズムが必要な人は別途やればよいと思います）。

## まとめ

- Vibe Coder の弱点は「作れない」ことではなく「**説明できない**」こと。
- 自分のAI生成コードに数問答えるだけで、基礎の穴が可視化される。
- 教科書は頭から読むより、**自分のコードの逆引き辞書**として使うと、退屈せず定着しやすい。

動くものを作れるのは、もう半分の強み。残り半分（説明できる）を、自分の作ったもので埋めていこうと思います。同じ「動くけど説明できない」状態の人がいたら、まずは自分のリポジトリを1ファイル開いて、自分に問いを出してみるのがおすすめです。
