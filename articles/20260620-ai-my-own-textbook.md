## はじめに

プログラミングの基礎をやり直そうと、入門書や競技プログラミングのサイトに何度か手を出しました。が、**続きません**。理由はだいたい同じで、「自分の目的（仕事でPythonを使えるようになりたい）とズレた内容」を「他人のペース」で大量に渡されると、急に手が止まる。

そこで方針を変えて、**AI（Claude Code のサブエージェントと Codex CLI）に、自分専用の教科書そのものを作らせる**ことにしました。この記事は、その作り方と、作る過程で効いた設計判断のメモです。

## この記事でわかること

- 市販の入門書が続かない人向けの「**自分専用教科書**」という選択肢
- AIに教科書を**書かせて、さらに監査までさせる**作り方（PASSするまで回す）
- 競プロ寄りの内容を**実務寄りに作り直した**話と、その判断基準

## なぜ「自分専用の教科書」なのか

自分はいわゆる Vibe Coder で、AIに指示すれば動くものは作れるけれど、各行を説明できない、という状態でした。足りないのは「読めて・直せて・説明できる」基礎。

ただ、その基礎を**汎用の入門書**で埋めようとすると、

- 自分の目的に対して**範囲が広すぎる/ズレている**（競プロのアルゴリズムなど）
- 章が「他人の順番」なので、どこまでやれば次へ進んでいいか分からない
- 1ページの情報量が多くて、ADHD気味の自分には**1日のゴールが曖昧**

という壁に毎回ぶつかっていました。だったら、**目的と特性に合わせた教科書を自分で持てばいい**。AIはこれが得意です。

## 作ったもの（教科書の構成）

Markdownで章ごとに作り、全章を**同じ固定フォーマット**に統一しました。

- 🎯 この章のゴール（1〜3個。終わったら何ができるか）
- 📖 解説（短い説明＋すぐ動くコード）
- ✍️ 例題（解答つき。まず写経）
- 🏋️ 演習（解答は別ファイル）
- 🌐 外部リソース課題（公式Docsなど）
- 🤖 AIへの振り分け（どこを自力でやり、どこをAIに任せるか）
- ✅ 章末チェックリスト
- 🧠 つまずき対策（特性に合わせたTips）

ポイントは **「読む本」ではなく「やる本」** にしたこと。毎章「何を・どの順で・どこまでやれば次へ進んでいいか」を明示しています。範囲も、競プロではなく**仕事で書くPython**（データ集計、自動化スクリプト、外部API、文字コードなど）に絞りました。

仕上げに `build_html.py` で全Markdownを1枚の「本」風HTMLにまとめ、マーカーや付箋を付けられるようにしています（手元で読むため）。

## 作り方: AIに書かせて、AIに監査させる

ただ書かせるだけだと品質がブレるので、**執筆と監査を分けた**のが肝でした。

1. **執筆**: Claude のサブエージェントと Codex CLI に章を書かせる。Codex はファイルを直接書けるので、長文の生成はこちらに寄せてトークンを節約。
2. **監査**: 別のエージェント（Claude サブエージェント）と Codex に、**「この教科書に従えば、自分の目的に到達できるか」**という観点で監査させる。Critical / Major / Minor に分類し、`PASS / FAIL` を出させる。
3. **修正 → 再監査** を、`Critical 0` で両者 PASS になるまで繰り返す。

独立した2つの監査者を**クロスチェック**に使うのがコツでした。実際、片方が見落とした不整合をもう片方が拾います。

## 効いた設計判断

### 競プロ向けを実務向けに作り直した

最初の版は、いま思うと**競技プログラミングに寄りすぎ**ていました。標準入力の暗記、全探索、bit全探索、BFS、DP……。これは「アルゴリズム試験」には効くけれど、自分の目的（業務でPythonを使う）からはズレている。

そこで第1〜4章を実務向けに全面的に作り直しました。

- 標準入力の暗記 → **`str` と `bytes`・文字コード**、コマンドライン引数・環境変数・ファイル入出力
- 計算量・全探索・累積和 → **内包表記・辞書/Counterでの集計・CSV/JSON**
- ABC攻略・DP → **自動化スクリプト・外部API（requests）・例外/リトライ・`pathlib`**

判断基準はシンプルで、「**自分が実際に書くコードに出てくるか**」。アルゴリズムのパズルは楽しいけれど、目的が違うなら主役にしない、と割り切りました（必要な人は別途やればいい、というスタンスです）。

### 監査が実際に不整合を捕まえた

作り直しのとき、**章本体の演習と、解答ファイルの問題が食い違う**ミスをしました（本体は新しい実務問題、解答は古い競プロ問題のまま）。これは監査が `Critical` として検出してくれて、危うく「答えの無い演習」を残すところでした。AIに大量編集させると、こういう**取り残し**が必ず出ます。監査ループはそのための保険でした。

### 教科書は「逆引き辞書」として使う

そして一番効いたのは、教科書を頭から読むのではなく、**自分のコードと往復**させる使い方です。

1. 教科書で概念を1つ学ぶ（例: `str` と `bytes`）
2. 自分のリポジトリでその概念が使われている所を探す
3. その部分を**AIなしで小さく再実装**する

教材が「自分の作ったもの」なので、退屈せず、理解できると素直に嬉しい。基礎の穴も「自分のコードに数問答える」だけで可視化できました。

## ハマりどころ

- **AIに丸投げすると黒箱が増える。** だから「監査」と「自分で再実装」をセットにしないと、説明できないコード/文章が積み上がるだけになる。
- **AIは自分の目的を知らない。** 放っておくと一般論・競プロ寄りになる。**目的を1ファイルに明文化**して、それに照らして監査させると精度が上がった。
- **Windows環境のAI生成物は地味に事故る。** 文字コード（cp932 / UTF-16）や改行（CRLF）絡みの不整合に何度か遭遇。実は今回も、Zennの記事を検証する手元スクリプトが CRLF を弾いて誤判定する、という小さな罠を踏みました。

## まとめ

- 入門書が続かないなら、**目的と特性に合わせた「自分専用教科書」をAIに作らせる**のは有効な選択肢。
- 鍵は「書かせる」だけでなく「**目的に照らして監査させ、PASSまで回す**」こと。独立した2つの監査者でクロスチェックすると取り残しを拾える。
- できた教科書は、**自分のコードの逆引き辞書**として使うと一番効く。

「動くものを作れる」のは半分の強み。残り半分（説明できる）を、自分で作った教材と自分のコードで埋めていこうと思います。
